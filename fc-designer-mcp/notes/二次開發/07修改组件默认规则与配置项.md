# 修改元件預設規則與配置項 ​

內建元件（如 `input`、`select`、`upload` 等）在設計器裡“拖入後生成什麼預設 rule、右側屬性面板顯示哪些配置項”，本質都由 **DragRule（拖拽規則）** 決定。本文說明如何在原始碼層二開：**修改內建元件的預設規則（rule）** 與 **右側 props 配置項的渲染規則（props 面板）**。

如果你還不熟悉 DragRule 的整體鏈路，建議先閱讀：[新增新元件](https://pro.form-create.com/doc/development/component)（裡面對 DragRule、渲染器註冊、設計態註冊的分層解釋最完整）。

## 你要改哪些地方？ ​

按影響範圍從小到大排序，通常有兩類改法：

- **僅改某個內建元件**

- 改預設 rule：編輯 `src/config/rule/<name>.js` 中的 `rule()` 方法返回值
- 改右側 props 面板：編輯同檔案中的 `props()` 方法返回的配置項陣列
- **改所有元件共享的基礎面板（影響全域）**

- 改欄位/標題/提示等：`src/config/base/field.js`
- 改表單級配置：`src/config/base/form.js`
- 改樣式面板：`src/config/base/style.js`
- 改校驗面板：`src/config/base/validate.js`
- 改高階（computed）面板：`src/config/base/advanced.js`

重要提醒

修改預設 rule（`DragRule.rule()`）**只對“新拖入元件”生效**。已儲存的歷史規則不會自動補齊/回填，需要單獨做遷移（見下文“相容性與遷移說明”）。

## DragRule 結構速覽 ​

一個典型的內建元件 DragRule（以 `input` 為例）通常長這樣：

js

```
export default {
  menu: 'main',
  icon: 'icon-input',
  label: '输入框',
  name: 'input',              // 关键：对应 rule.type
  input: true,
  event: ['blur', 'focus'],
  validate: ['string', 'email'],

  // 1) 默认规则：拖入后生成的 rule
  rule({ t }) {
    return {
      type: 'input',
      field: uniqueId(),
      title: t('com.input.name'),
      info: '',
      $required: false,
      props: {},
    };
  },

  // 2) 右侧 props 面板：显示哪些配置项、怎么渲染
  props(_, { t }) {
    return localeProps(t, 'input.props', [
      { type: 'switch', field: 'disabled' },
      { type: 'input', field: 'placeholder' },
      // ...
    ]);
  },
};
```

你這篇文件要做的改動，基本都集中在 `rule()` 與 `props()` 兩個方法裡。

## 修改內建元件的預設規則 ​

預設規則的核心目標是：**讓“新拖入的元件”帶著你想要的初始配置**，例如預設 `placeholder`、預設 `clearable: true`、預設上傳 action、預設 options 等。

### 1. 定位 DragRule 檔案 ​

內建元件規則都在：

- `src/config/rule/<name>.js`

其中 `<name>` 通常等於 DragRule 的 `name`（也就是 `rule.type`）。

例如：

- 輸入框：`src/config/rule/input.js`
- 選擇器：`src/config/rule/select.js`
- 上傳：`src/config/rule/upload.js`

### 2. 修改 `rule()` 返回的預設 rule ​

以 `input` 為例：如果你想讓新拖入的輸入框預設帶佔位和可清空按鈕，可以這樣改：

js

```
rule({ t }) {
  return {
    type: name,
    field: uniqueId(),
    title: t('com.input.name'),
    info: '',
    $required: false,
    props: {
      placeholder: t('com.input.props.placeholder') || '请输入',
      clearable: true,
    },
  };
}
```

以 `upload` 為例：很多專案希望預設 `listType`、`action`、`onSuccess` 都有“可直接跑通”的值，則在 `rule()` 裡設定 `props` 預設值即可（專案內 `upload` 就是這種寫法）。

建議

- **預設值儘量放在 `rule().props` 裡**，讓“預設行為”與“右側 props 面板修改的欄位”保持一致。
- 如果某個預設值會影響渲染結構（例如 select 的 options、layout 的 children），建議同時檢查該元件是否存在 `watch`、`loadRule/parseRule` 等特殊處理邏輯，避免預設值與轉換邏輯衝突。

## 修改右側配置項 ​

`props()` 返回的是一個“配置表單的規則陣列”，每一項描述一個配置控制元件如何渲染、繫結到哪個欄位、有哪些選項/提示/校驗等。

### 1. `field` 的繫結規則（非常關鍵） ​

在 `DragRule.props()` 中：

- **預設繫結目標是 `rule.props`**  
 例如 `{ field: 'disabled' }` 表示繫結到 `rule.props.disabled`。

在 `base/*.js`（通用面板）裡：

- `field` 往往直接指向 **rule 頂層欄位** 或一個 **巢狀路徑**  
 例如 `field: 'field'`（rule.field），以及 `field: 'wrap>style>marginBottom'` 這種路徑寫法。
- 也存在使用點號路徑的情況（例如高階面板中 `field: 'props.disabled'`），用於直接定位到 `rule.props.disabled`。

> 實戰建議：在 `rule/*.js` 裡優先用“相對 props”的寫法（`disabled`、`placeholder`）；在 `base/*.js` 裡用“完整路徑”的寫法（`wrap>style>...` / `props.xxx`）。

### 2. 常見配置項結構（你能改什麼） ​

一個配置項通常包含這些欄位（按常見程度排序）：

- **`type`**：配置控制元件型別（如 `switch`、`input`、`select`、`inputNumber`、`FnInput`、`TableOptions` 等）
- **`field`**：繫結欄位（見上文）
- **`title`**：標題（多語言時通常由 `localeProps` 自動補）
- **`value`**：預設值（僅對“配置面板首次生成/回填”有意義，是否生效依賴設計器實現）
- **`props`**：傳給該配置控制元件元件的引數（例如 slider 的 `min/max`）
- **`options`**：select/radio 等元件的選項（常配合 `localeOptions`）
- **`warning`**：警告/提示文案
- **`children`**：複雜配置項的子項（例如 `ConfigItem` 容器包裹）

### 3. 示例：給 `input` 增加一個右側配置項 ​

假設你想在 `input` 的 props 面板裡新增 `showPassword`（僅當 `type=password` 時有意義，這裡先給通用開關示例），在 `src/config/rule/input.js` 的 `props()` 陣列裡追加：

js

```
props(_, { t }) {
  return localeProps(t, name + '.props', [
    { type: 'switch', field: 'disabled' },
    { type: 'switch', field: 'readonly' },
    // ...原有项

    // 新增：显示密码切换按钮（业务自定义示例）
    { type: 'switch', field: 'showPassword' },
  ]);
}
```

如果你需要一個帶選項的配置項（例如 input 的 `type`），參考專案內寫法：

js

```
{
  type: 'select',
  field: 'type',
  options: localeOptions(t, [
    { label: 'text', value: 'text' },
    { label: 'number', value: 'number' },
    { label: 'date', value: 'date' },
  ]),
}
```

### 4. 示例：調整 `select` 的 options / 遠端配置能力 ​

`select` 是典型的“既要 options，又可能要 remoteMethod”的元件。專案內已經提供了：

- `makeOptionsRule(t, 'options')`：生成 options 配置項
- `FnInput` + `getInjectArg(t)`：生成函式輸入配置（remoteMethod）

你可以用同樣方式“改預設 options 結構”或“新增/刪減遠端相關項”：

js

```
return localeProps(t, name + '.props', [
  makeOptionsRule(t, 'options'),
  { type: 'switch', field: 'filterable' },
  { type: 'switch', field: 'remote' },
  {
    type: 'FnInput',
    field: 'remoteMethod',
    props: {
      body: true,
      button: true,
      fnx: true,
      name: 'remoteMethod',
      args: [getInjectArg(t)],
    },
  },
]);
```

常見坑

右側配置項“加了但沒效果”，最常見原因是：

- 配置項 `field` 寫錯（你以為寫到了 `rule.props.xxx`，但實際繫結到別處）
- 執行態元件並沒有消費這個 prop（比如元件內部沒讀 `props.showPassword`）
- 多語言 key 缺失導致標題/提示為空（尤其當專案強制依賴 `localeProps` 時）

## 修改通用基礎配置 ​

如果你要修改的是“所有元件都共有的面板項”，例如：

- 欄位 ID、標題、提示
- form-item 佈局（labelWidth、span、margin）
- 全域樣式項（id/class/style）
- 全域校驗/高階 computed 項

那麼應該改 `src/config/base/*.js`。

### 示例：在“樣式”面板增加一個全域配置項 ​

`src/config/base/style.js` 本質上返回一組配置項（示例中已有 `id/class/style`）。你可以在這裡新增你的全域配置元件/輸入項。

### 示例：在“表單配置”裡增加一個視覺化配置塊 ​

`src/config/base/form.js` 裡大量使用 `ConfigItem` 容器來組織複雜配置，並用 `update()` 回撥把“是否已配置”的資訊寫回 UI（例如給配置塊顯示 `info`）。

如果你想新增一個“高階配置塊”，可以複用相同模式：

- 用 `ConfigItem` 包裹
- 內部 children 用 `StyleConfig / TableOptions / FnConfig` 等配置元件
- 需要“已配置提示”時，參考 `updateConfigInfo(...)` 寫法

## 常見問題排查 ​

### 1. 改了 `rule()` 但拖入還是舊預設值 ​

- **現象**：新拖入元件仍然是舊的預設 props
- **排查**： 

- 是否改錯了檔案（確認 `src/config/rule/<name>.js` 與 `name` 匹配）
- 是否存在多入口/多構建目標，實際執行引用的是另一份配置目錄
- 是否有快取產物未重新整理（重啟 dev、清快取）

### 2. 右側配置項顯示了，但改動不影響元件 ​

- **最可能原因**：執行態元件未消費對應 prop，或該 prop 名在 UI 框架裡不同（例如 Element Plus 與 AntDV 的 prop 命名差異）
- **排查**： 

- 對照 UI 元件文件確認 prop 名是否正確
- 在執行態渲染元件中查詢該 prop 是否被讀取/透傳

### 3. 歷史表單不生效 ​

- **原因**：預設 rule 只對新拖入生效
- **處理**：見下文“相容性與遷移說明”

## 相容性與遷移說明 ​

當你修改了預設 rule 或新增了 props 欄位：

- **對新拖入元件**：立即生效
- **對歷史儲存規則**：不會自動補齊欄位（仍然是舊資料）
