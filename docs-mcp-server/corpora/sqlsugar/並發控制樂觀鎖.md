## 速查表

| 需求 | 語法 |
|---|---|
| 安全累加(不查庫) | Updateable<T>().SetColumns(it=>it.Num==it.Num+1).Where(...) |
| 樂觀鎖(版本欄位) | [SugarColumn(IsEnableUpdateVersionValidation=true)] |
| 版本不符 | 拋 VersionExceptions,需攔截重試 |
| Timestamp 版(SqlServer) | 版本欄位用 timestamp/byte[] |
| 悲觀鎖 | With(SqlWith.UpdLock) + 事務 |
| 一對多提交 | 事務內逐表更新 |

---

## 三種並發場景與對應方案

| 場景 | 問題 | 解法 |
|---|---|---|
| 數值累加（點贊、扣款）| 先讀後寫造成丟失更新 | SET num = num + 1（資料庫層面累加）|
| 編輯表單提交覆蓋 | A 和 B 同時打開同一筆資料，後者覆蓋前者 | 樂觀鎖（版本號比對）|
| 讀取後立即插入 | 兩個請求同時判斷「不存在」後都插入 | 悲觀鎖（SELECT FOR UPDATE）|

## 一、安全累加（不查庫直接 SET）

```csharp
// ✅ 正確：SET num = num + 1（資料庫層面，並發安全）
db.Updateable<Order>()
    .SetColumns(it => it.Num == it.Num + 1)
    .Where(it => it.Id == 1)
    .ExecuteCommand();
// → UPDATE Order SET Num = Num + 1 WHERE Id = 1

// ❌ 錯誤：先在 C# 計算再寫回（並發不安全，可能丟失更新）
var num = data.Num + 1;
db.Updateable<Order>()
    .SetColumns(it => it.Num == num)   // 兩個並發請求可能都讀到同一個值
    .Where(it => it.Id == 1)
    .ExecuteCommand();

// 批量累加（每筆各自 +1）
db.Updateable(list)
    .PublicSetColumns(it => it.Num, it => it.Num + 1)
    .ExecuteCommand();

// 批量累加（+list 中對應的值）
db.Updateable(list)
    .PublicSetColumns(it => it.Price, "+")   // SET price = price + list[i].price
    .ExecuteCommand();
```

## 二、樂觀鎖（防止提交覆蓋）

### 適用場景

使用者打開編輯頁面 → 填寫資料 → 提交儲存。
若另一個使用者在此期間已修改同一筆資料，後者提交時應報錯或靜默失敗，而非直接覆蓋。

### 方式一：不依賴資料庫自動更新（推薦新專案）

版本欄位支援類型：long、string、Guid、DateTime（不推薦 DateTime，有精度問題）。
前端傳 long 時注意精度丟失，建議序列化成 string 傳遞。

```csharp
// 實體設定
public class Order
{
    [SugarColumn(IsPrimaryKey = true, IsIdentity = true)]
    public int Id { get; set; }
    public string Name { get; set; }

    [SugarColumn(IsEnableUpdateVersionValidation = true)]  // 標記為版本欄位
    public long Ver { get; set; }  // 建議用 string 或 Guid，long 前端有精度問題
}
```

```csharp
// 用法一：版本不符合返回 0，不拋例外
int rows = db.Updateable(new Order
{
    Id   = id,
    Name = "新名稱",
    Ver  = frontendVer   // 前端傳來的版本號，更新成功後會自動更新 Ver
}).ExecuteCommandWithOptLock();

if (rows == 0)
    throw new Exception("資料已被他人修改，請重新整理後再試");

// 用法二：版本不符合直接拋 VersionExceptions
try
{
    db.Updateable(new Order
    {
        Id   = id,
        Name = "新名稱",
        Ver  = frontendVer
    }).ExecuteCommandWithOptLock(true);  // true = 版本不符合時拋例外
}
catch (VersionExceptions)
{
    // 並發衝突
    throw new Exception("資料已被他人修改，請重新整理後再試");
}
```

**✅ 完整流程**

```csharp
// 1. 查詢時把版本號連同資料一起回傳前端
var order = db.Queryable<Order>().Where(o => o.Id == id).First();
// 回傳 { id: 1, name: "舊名稱", ver: 1551128313597136896 }

// 2. 前端儲存時把 ver 帶回來
// PUT /api/order  { id: 1, name: "新名稱", ver: 1551128313597136896 }

// 3. 後端更新時比對版本
int rows = db.Updateable(new Order
{
    Id   = req.Id,
    Name = req.Name,
    Ver  = req.Ver     // 比對資料庫中的 Ver，若不同則更新失敗
}).ExecuteCommandWithOptLock();

if (rows == 0)
    return ApiResult.Fail("資料已被他人修改，請重新整理後再試");
```

### 方式二：依賴資料庫自動更新（Timestamp，適合 SqlServer）

| 資料庫 | DB 欄位類型 | C# 類型 |
|---|---|---|
| SqlServer | timestamp（自動更新）| byte[] |
| 其他資料庫 | timestamp（需設自動更新）| DateTime |

```csharp
// SqlServer 實體設定
public class Order
{
    [SugarColumn(IsPrimaryKey = true, IsIdentity = true)]
    public int Id { get; set; }
    public string Name { get; set; }

    [SugarColumn(
        IsEnableUpdateVersionValidation = true,  // 標記版本欄位
        IsOnlyIgnoreInsert = true,               // 插入時不插入（由 DB 自動產生）
        IsOnlyIgnoreUpdate = true,               // 更新時不更新（由 DB 自動更新）
        ColumnDataType = "timestamp"
    )]
    public byte[] Timestamp { get; set; }  // SqlServer 用 byte[]，其他 DB 用 DateTime
}
```

```csharp
// 查詢（Timestamp 會一起帶出）
var data = db.Queryable<Order>().InSingle(id);

// 更新時啟用版本驗證（Timestamp 必須與 DB 中的值相同才能更新）
db.Updateable(data).IsEnableUpdateVersionValidation().ExecuteCommand();
// 成功 → DB 中 Timestamp 自動更新
// 失敗（別人已更新）→ 拋出 VersionExceptions
```

```csharp
// 捕獲並發錯誤
try
{
    db.Updateable(data).IsEnableUpdateVersionValidation().ExecuteCommand();
}
catch (VersionExceptions)
{
    // 確定是版本衝突（並發問題）
    throw new Exception("資料已被他人修改");
}
```

## 三、一對多表單提交

一對多時只需對主表做樂觀鎖，子表無需額外處理（子表和主表一起提交，本身就是原子操作）：

```csharp
int rows = db.Updateable(order).ExecuteCommandWithOptLock();
if (rows == 0)
    return ApiResult.Fail("資料已被他人修改");

// 子表整批替換（不需要版本控制）
db.Deleteable<OrderItem>().Where(o => o.OrderId == order.Id).ExecuteCommand();
db.Insertable(order.Items).ExecuteCommand();
```

## 四、悲觀鎖（SELECT FOR UPDATE）

適合「讀取後立即插入/更新」的場景，防止兩個並發請求同時通過「不存在」判斷後都執行操作。

```csharp
// 詳見事務鎖文件（typeId=2416）
// 在事務內搭配 FOR UPDATE 使用
db.Ado.BeginTran();
var data = db.Queryable<Order>()
    .Where(o => o.OrderNo == orderNo)
    .With(SqlWith.UpdLock)   // FOR UPDATE / WITH (UPDLOCK)
    .FirstOrDefault();

if (data == null)
    db.Insertable(newOrder).ExecuteCommand();
else
    db.Updateable(data).ExecuteCommand();

db.Ado.CommitTran();
```

## 兩種樂觀鎖比較

| 比較項目 | 方式一（不依賴 DB 自動更新）| 方式二（依賴 DB Timestamp）|
|---|---|---|
| 適用場景 | 新專案，通用 | SqlServer timestamp 欄位 |
| 版本類型 | long / string / Guid | byte[]（SqlServer）/ DateTime |
| 前端傳值 | 需帶 Ver 回來 | 需帶 Timestamp 回來 |
| 執行方法 | ExecuteCommandWithOptLock() | IsEnableUpdateVersionValidation() |
| 版本自動更新 | SqlSugar 自動更新 | DB 自動更新 |
| 推薦程度 | 推薦（新功能）| 舊功能，已有 timestamp 欄位時使用 |

---

## 注意事項

- 樂觀鎖只能用在單筆實體更新，不能用在集合批量更新
- long 類型版本號傳給前端時建議序列化成 string，避免 JavaScript 精度丟失
- Timestamp（byte[]）由 DB 自動管理，C# 端設 IsOnlyIgnoreInsert 和 IsOnlyIgnoreUpdate
- 並發衝突拋出的是 VersionExceptions，可以精確捕獲區分並發錯誤和其他錯誤
- 悲觀鎖適合「查了就要用」的場景，樂觀鎖適合「打開頁面後才提交」的場景
