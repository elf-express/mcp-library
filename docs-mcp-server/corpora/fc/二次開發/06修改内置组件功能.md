# 修改內建元件功能 ​

在二開場景中，你經常會遇到“內建元件差一點就滿足需求”的情況：需要加一個互動、補一個事件、改一下回顯邏輯、或者相容歷史資料格式。本文介紹在設計器中，如何對“內建元件”進行功能改造，並給出兩種內建元件的推薦處理方式。

## 你要改哪些地方？ ​

“修改內建元件功能”通常涉及 3 層（按需選擇）：

- **元件實現（Runtime Component）**：真正決定互動、v-model、事件與展示的 Vue 元件
- **規則/配置（DragRule，可選）**：如果你新增/修改了可配置項，需要同步更新對應的規則定義與多語言
- **註冊與覆蓋（僅渲染器內建元件）**：不改渲染器原始碼，透過“同名元件覆蓋”的方式替換內建實現

下面分兩類說明。

## 一、修改設計器內建元件 ​

設計器內建元件屬於設計器專案自身的一部分，通常可以直接在元件檔案內完成改造，成本最低、定位最直觀。

### 參考位置（以 SignaturePad 為例） ​

- **PC 元件**：`/src/components/SignaturePad.vue`
- **Mobile 元件**：`/src/components/mobile/SignaturePad.vue`

### 實施步驟 ​

### 1. 備份預設程式碼 ​

建議用“最輕量、可長期維護”的備份方式之一：

- **方式 A：同目錄備份檔案**（最快） 

- 例如：`SignaturePad.vue.bak` 或 `SignaturePad.vue.default`
- 同時建議把 `*.bak` / `*.default` 加入專案的 `.gitignore`（避免誤提交）
- **方式 B：用 Git 儲存一份“原版快照”**（最推薦） 

- 建議在修改前保留一次提交或打 tag，後續升級/回滾最省事

WARNING

“備份”不是為了多一份檔案，而是為了在升級設計器版本、對比差異、回滾問題時，有一個可追溯的基線。

### 2. 直接修改元件實現 (SignaturePad.vue) ​

常見改造點（按需選擇）：

- **互動邏輯**：比如新增按鈕、增加快捷操作、修復邊界輸入
- **資料格式**：調整輸出值格式（保持向後相容，見下文“相容建議”）
- **事件丟擲**：補齊 `change`、自定義事件（例如 `clear`、`confirm`）
- **只讀/閱讀模式**：根據 `disabled` 或 `formCreateInject.preview` 提供更合理的回顯
- **跨端一致性**：如果同時支援 PC/Mobile，建議兩端保持同名同契約

### 3. 如有新增配置項，同步更新 DragRule ​

如果你的功能改造引入了新的 `props.xxx`、新增了右側面板配置項，需要同步改 DragRule：

- 對應規則檔案一般在 `/src/config/rule/*`
- 同步補齊多語言（如 `/src/locale/*`）

否則會出現“元件支援了新能力，但設計器右側無法配置”的割裂體驗。

## 二、修改渲染器內建元件 ​

渲染器內建元件存在於渲染器包（例如 Element Plus / Ant Design Vue / Vant 的渲染器實現）中，**不推薦直接修改渲染器原始碼**：

- 升級成本高：一升級就容易丟改動/產生衝突
- 影響面大：改動會對所有使用方產生全域影響，排查困難

因此更推薦使用“**開發一個同名元件並掛載**”的方式覆蓋內建元件。

### 原理：同名註冊會覆蓋內建實現 ​

form-create 允許透過 `formCreate.component(key, Comp)` 註冊元件；當 `key` 與內建元件同名時，後註冊的元件會覆蓋內建元件。

> 更通用的說明可參考文件：[覆蓋內建元件](https://pro.form-create.com/doc/override-component)

### 實施步驟 ​

### 1. 確認你要覆蓋的元件 key（必須一致） ​

你需要知道該內建元件在規則裡使用的 `type`（也就是註冊 key），例如：`input`、`select`、`radio` 等。

### 2. 開發一個“同名元件” ​

開發一個自定義元件，並儘量保持與原元件**功能相容**（非常關鍵）：

- **v-model**：`modelValue` + `update:modelValue`（Vue2 則為 `value` + `input`）
- **disabled**：支援禁用
- **change**：建議透出 `change`（便於統一監聽）
- **attrs 透傳**：`v-bind="$attrs"`，避免丟失渲染器/規則傳入的原生屬性
- **插槽/事件相容（按需）**：儘量保持常用 slot、事件名不變

### 3. 在應用啟動階段完成覆蓋註冊（越早越好） ​

在建立任何表單例項之前執行覆蓋註冊，例如：

js

```
import formCreate from '@form-create/element-ui';
import CustomInput from './CustomInput.vue';

// 覆盖内置 input
formCreate.component('input', CustomInput);
```

## 常見問題排查 ​

### 1. 覆蓋渲染器內建元件後，部分表單配置失效 ​

- **最可能原因**：自定義元件沒有 `v-bind="$attrs"`，導致規則傳入的 props/事件丟失
- **修復建議**：補齊 attrs 透傳，並檢查 `emits` 是否包含必要事件

### 2. 歷史表單出現異常（值型別不相容/回顯錯誤） ​

- **最可能原因**：你修改了輸出值格式，但沒有相容舊格式
- **修復建議**：在元件內部對舊值做相容解析，或提供“遷移指令碼/轉換函式”

## 相容性建議 ​

- **儘量不改 `rule.type`**：改 type 會直接影響歷史規則渲染
- **新增能力優先“向後相容”**：舊配置仍能正常工作，新配置只是增強
- **覆蓋渲染器元件時儘量保持 API 不變**：屬性名、事件名、插槽儘量沿用原元件
- **跨端同名同契約**：PC/Mobile 最好保持一致，避免同一份規則跨端表現不一致
