## 問題根源

ORM 產生的 SQL 參數編碼與資料庫欄位類型不一致時，**索引可能失效**，導致查詢變慢。

| 資料庫 | 常見問題 | ORM 預設 |
|---|---|---|
| SQL Server | 欄位是 `varchar`，ORM 參數是 `N'xxx'`（nvarchar）→ 索引失效 | nvarchar |
| Oracle | 欄位是 `Date`，ORM 參數是 `DateTime` → 索引失效 | DateTime |

---

## 一、如何確認真實 SQL

```csharp
// AOP 中輸出真實 SQL（含參數化後的完整 SQL）
db.Aop.OnLogExecuting = (sql, pars) =>
{
    // GetNativeSql 會把參數值直接嵌入 SQL，就是資料庫實際執行的樣子
    Console.WriteLine(UtilMethods.GetNativeSql(sql, pars));
};

// 也可以用 SqlServer Profiler 監控真實執行 SQL 和時間
// 如果 Profiler 裡的 SQL 快，但 ORM 感覺慢 → 問題不在 SQL，看「其他情況優化」章節
```

## 二、解決索引失效（新版，5.0.8.6+，推薦）

### 2.1 欄位特性指定類型

```csharp
// SQL Server：欄位是 varchar → 指定參數為 AnsiString
[SugarColumn(SqlParameterDbType = System.Data.DbType.AnsiString)]
public string OrderNo { get; set; }
// 產生 SQL：WHERE OrderNo = 'A001'（不加 N 前綴）✅

// SQL Server：欄位是 char(10)
[SugarColumn(SqlParameterDbType = System.Data.DbType.AnsiStringFixedLength)]
public string Code { get; set; }

// SQL Server：指定長度（SqlParameterDbType 有值時 SqlParameterSize 才生效）
[SugarColumn(SqlParameterDbType = System.Data.DbType.AnsiString, SqlParameterSize = 50)]
public string ShipmentNo { get; set; }

// Oracle：欄位是 Date → 指定參數為 Date（去掉時分秒）
[SugarColumn(SqlParameterDbType = System.Data.DbType.Date)]
public DateTime InvoiceDate { get; set; }
// 產生 SQL：WHERE InvoiceDate = TO_DATE('2024-01-01', 'YYYY-MM-DD') ✅
```

### 2.2 DbType 類型對照表

| DbType 值 | 對應 SQL 類型 | 適用場景 |
|---|---|---|
| `AnsiString` | varchar | SQL Server varchar 欄位（最常用）|
| `AnsiStringFixedLength` | char | SQL Server char 欄位 |
| `String` | nvarchar | SQL Server nvarchar（ORM 預設）|
| `StringFixedLength` | nchar | SQL Server nchar 欄位 |
| `Date` | 短日期（無時分秒）| Oracle Date、部分日期欄位 |
| `DateTime` | 完整日期時間 | 一般 DateTime（ORM 預設）|

### 2.3 原生 SQL 指定類型

```csharp
// 手寫 SQL 時也可以指定參數類型
var param = new SugarParameter("@orderNo", "A001", System.Data.DbType.AnsiString);
var list  = db.Ado.SqlQuery<Order>(
    "SELECT * FROM [Order] WHERE OrderNo = @orderNo", param);
```

## 三、解決索引失效（老版本）

```csharp
// 方式 1：全局關閉 nvarchar（所有表的字串參數都改成 varchar）
// 適合所有表都沒有 nvarchar 欄位的情況（如純 varchar 的舊系統）
MoreSettings = new ConnMoreSettings
{
    DisableNvarchar = true  // 所有字串參數從 N'xxx' 改成 'xxx'
}

// 方式 2：當前方法有效（只影響這個方法的查詢）
db.CurrentConnectionConfig.MoreSettings = new ConnMoreSettings { DisableNvarchar = true };
var list = db.Queryable<Order>().Where(it => it.OrderNo == "A001").ToList();

// 方式 3：指定特定條件（只有這個條件改用 varchar）
var list = db.Queryable<Order>()
    .Where(it => it.OrderNo == SqlFunc.ToVarchar("A001"))
    .ToList();
// ⚠️ 函數只能用在參數上，走索引的欄位禁止套函數

// 方式 4：原生 SQL 指定類型
var param = new SugarParameter("@name", "張三", System.Data.DbType.AnsiString);
```

## 四、AIRSET 實際場景（Chinese_Taiwan_Stroke_CI_AS Collation）

AIRSET 資料庫使用 `Chinese_Taiwan_Stroke_CI_AS` Collation，
舊有的 `varchar`/`nvarchar` 混用欄位特別容易出現索引失效問題：

```csharp
// 出貨單號（varchar）
[SugarTable("Inv1發票收據")]
public class Invoice
{
    [SugarColumn(IsPrimaryKey = true)]
    public string 收據單號 { get; set; }

    // 如果欄位是 varchar，指定 AnsiString 避免索引失效
    [SugarColumn(SqlParameterDbType = System.Data.DbType.AnsiString, SqlParameterSize = 20)]
    public string 訂單編號 { get; set; }

    [SugarColumn(SqlParameterDbType = System.Data.DbType.AnsiString, SqlParameterSize = 20)]
    public string 客戶代號 { get; set; }
}

// 查詢時索引正常走
var list = db.Queryable<Invoice>()
    .Where(it => it.訂單編號 == orderNo)  // → WHERE 訂單編號 = 'A001'（不加 N）
    .ToList();
```

## 五、其他慢查詢排查

### 5.1 真實 SQL 快但 ORM 感覺慢

```
原因：問題不在 SQL，而在其他地方
排查方向：
1. 網路延遲（應用伺服器到資料庫的網路）
2. AOP 事件中有耗時操作（如 GetService、反射、日誌 IO）
3. 實體綁定（列多時略慢）→ 改用實體而非匿名物件
4. 二級快取操作（GetAllKey 掃描太多 Key）
```

```csharp
// AOP 中避免耗時操作
db.Aop.OnLogExecuting = (sql, pars) =>
{
    // ✅ 快：直接輸出
    Console.WriteLine(UtilMethods.GetNativeSql(sql, pars));

    // ❌ 慢：每次都 BuildServiceProvider（不要這樣用）
    // var svc = services.BuildServiceProvider().GetService<MyService>();
};
```

### 5.2 鎖或事務引起的慢

```csharp
// SQL Server 查詢加 NOLOCK（犧牲一致性換效能，適合報表/統計查詢）
var list = db.Queryable<Order>()
    .With(SqlWith.NoLock)    // → FROM [Order] WITH(NOLOCK)
    .Where(it => it.Status == 1)
    .ToList();

// 或在 OnExecutingChangeSql 中全局加 NOLOCK（謹慎使用）
db.Aop.OnExecutingChangeSql = (sql, pars) =>
{
    // 只對特定表加 NOLOCK（需自行判斷）
    return new KeyValuePair<string, SugarParameter[]>(sql, pars);
};
```

### 5.3 用 SqlServer Profiler 定位慢 SQL

```
步驟：
1. 開啟 SQL Server Management Studio
2. 工具 → SQL Server Profiler
3. 連線到目標資料庫
4. 篩選 Duration > 1000ms（超過 1 秒的 SQL）
5. 找到慢 SQL 後複製到 SSMS 按 F5 執行，查看執行計畫
6. 看執行計畫中是否有「Index Scan」→ 應改為「Index Seek」
7. 確認參數類型是否與欄位類型一致（N'' vs ''）
```

## 注意事項

- `SqlParameterDbType` 只在 Lambda 表達式的條件中生效，不影響其他場景
- 欄位套函數（如 `CONVERT`、`UPPER`）**一定會讓索引失效**，條件欄位禁止套函數
- `DisableNvarchar = true` 是全局設定，確認所有表的字串欄位都是 varchar 才能用
- AIRSET 的 `Chinese_Taiwan_Stroke_CI_AS` Collation 是大小寫不分、筆畫排序，varchar 和 nvarchar 混用特別容易索引失效
- 只有「參數類型」和「欄位類型」不一致時才會失效，如果資料庫欄位本來就是 nvarchar，ORM 預設的 nvarchar 參數就沒問題
