## 速查表

| 需求 | 語法 |
|---|---|
| 悲觀鎖等待 | TranLock(DbLockType.Wait) |
| 悲觀鎖排它(報錯) | TranLock(DbLockType.Error) |
| 前提 | 必須在事務內 BeginTran/CommitTran |
| 行鎖 vs 表鎖 | 條件命中索引才是行鎖,否則升級表鎖 |
| db 執行緒安全 | 高並發每請求 CopyNew / new |
| 樂觀 vs 悲觀 | 低衝突用樂觀;庫存/帳務用悲觀 |

## 三種並發控制方式比較

| 方式 | 語法 | 說明 | 適用場景 |
|---|---|---|---|
| 樂觀鎖 | [SugarColumn(IsEnableUpdateVersionValidation = true)] | 版本號衝突時拋例外 | 編輯表單、低衝突場景 |
| 悲觀鎖（等待）| TranLock(DbLockType.Wait) | 第一個執行，其餘等待 | 庫存扣減、帳務處理 |
| 悲觀鎖（排它）| TranLock(DbLockType.Error) | 第一個執行，其餘直接報錯 | 秒殺、單次執行業務 |

樂觀鎖詳細用法見：並發控制樂觀鎖.md（typeId=2399）

## 一、悲觀鎖 — 等待模式（DbLockType.Wait）

**行為**：兩個相同業務同時執行時，第一個先跑，第二個**等待**第一個完成後再執行。

```csharp
// 標準寫法（必須包在事務中）
db.BeginTran();
try
{
    // 查詢並鎖定記錄（盡量用主鍵條件，避免表鎖）
    var order = db.Queryable<Order>()
        .TranLock(DbLockType.Wait)      // 加悲觀鎖（等待模式）
        .Where(it => it.Id == orderId)
        .First();

    if (order == null || order.Status != 1)
        throw new Exception("訂單狀態不符");

    // 業務邏輯（此時其他事務若鎖定同一筆記錄，會在此等待）
    order.Status = 2;
    order.ProcessTime = DateTime.Now;
    db.Updateable(order).ExecuteCommand();

    db.CommitTran();
}
catch
{
    db.RollBackTran();
    throw;
}
```

**✅ 實際案例：銀行明細沖帳（AIRSET 帳務場景）**

```csharp
public async Task<bool> ProcessPaymentAsync(int accountId, decimal amount)
{
    db.BeginTran();
    try
    {
        // 鎖定帳戶餘額記錄（主鍵查詢 = 行鎖，不影響其他帳戶）
        var account = db.Queryable<Account>()
            .TranLock(DbLockType.Wait)
            .Where(it => it.Id == accountId)
            .First();

        if (account.Balance < amount)
            throw new Exception("餘額不足");

        // 扣款
        account.Balance -= amount;
        db.Updateable(account)
            .UpdateColumns(it => new { it.Balance })
            .ExecuteCommand();

        // 寫入明細
        db.Insertable(new AccountDetail
        {
            AccountId   = accountId,
            Amount      = -amount,
            Balance     = account.Balance,
            CreateTime  = DateTime.Now
        }).ExecuteCommand();

        db.CommitTran();
        return true;
    }
    catch
    {
        db.RollBackTran();
        return false;
    }
}
```

## 二、悲觀鎖 — 排它模式（DbLockType.Error）

**行為**：兩個相同業務同時執行時，只有第一個成功，其餘**立即拋出例外**（不等待）。

```csharp
db.BeginTran();
try
{
    // 查詢並鎖定（若已被鎖定則立即報錯）
    var inventory = db.Queryable<Inventory>()
        .TranLock(DbLockType.Error)     // 加悲觀鎖（排它模式）
        .Where(it => it.ProductId == productId)
        .First();

    if (inventory.Stock < quantity)
        throw new Exception("庫存不足");

    inventory.Stock -= quantity;
    db.Updateable(inventory)
        .UpdateColumns(it => new { it.Stock })
        .ExecuteCommand();

    db.CommitTran();
}
catch
{
    db.RollBackTran();
    throw;  // 業務層捕捉後回傳「搶購失敗」
}
```

## 三、重要注意事項

### 3.1 行鎖 vs 表鎖

```csharp
// ✅ 主鍵條件 → 行鎖（只鎖定單筆記錄，其他記錄不受影響）
db.Queryable<Order>()
    .TranLock(DbLockType.Wait)
    .Where(it => it.Id == 1)          // 主鍵查詢
    .First();

// ⚠️ 非主鍵條件 → 可能升級為表鎖（影響所有記錄）
db.Queryable<Order>()
    .TranLock(DbLockType.Wait)
    .Where(it => it.Status == 1)      // 非主鍵，視索引情況可能鎖全表
    .ToList();
```

**原則：TranLock 查詢條件盡量用主鍵，且返回筆數越少越好。**

### 3.2 db 實例執行緒安全

```csharp
// ✅ 多執行緒場景：每個 Task 必須 New 一個新的 db 實例
for (int i = 0; i < 10; i++)
{
    Task.Run(() =>
    {
        var db = GetInstance();  // 每次 new 一個，保證執行緒安全
        try
        {
            db.BeginTran();
            var data = db.Queryable<Order>()
                .TranLock(DbLockType.Error)
                .Where(it => it.Id == 1)
                .First();

            Thread.Sleep(1000);  // 模擬業務處理時間
            db.CommitTran();
            Console.WriteLine("成功");
        }
        catch (Exception ex)
        {
            db.RollBackTran();
            Console.WriteLine($"失敗：{ex.Message}");
        }
    });
}
// 結果：只有 1 個成功，其餘 9 個立即失敗（Error 模式）
```

### 3.3 必須有事務

```csharp
// ❌ 沒有事務，TranLock 不會生效
var data = db.Queryable<Order>()
    .TranLock(DbLockType.Wait)
    .Where(it => it.Id == 1)
    .First();

// ✅ 必須包在 BeginTran / CommitTran 中
db.BeginTran();
var data = db.Queryable<Order>()
    .TranLock(DbLockType.Wait)
    .Where(it => it.Id == 1)
    .First();
// ... 業務邏輯 ...
db.CommitTran();
```

## 四、樂觀鎖 vs 悲觀鎖 選擇指引

| 情境 | 推薦 |
|---|---|
| 多人同時編輯同一筆資料（表單場景）| 樂觀鎖 |
| 庫存扣減、餘額變更（需要絕對正確）| 悲觀鎖 Wait |
| 秒殺、唯一序號產生（只允許一人成功）| 悲觀鎖 Error |
| 高並發讀多寫少 | 樂觀鎖（效能好）|
| 高並發寫多，業務必須等待 | 悲觀鎖 Wait |
| 高並發寫多，失敗直接報錯 | 悲觀鎖 Error |

---

## 注意事項

- 需要 SqlSugar 5.0.8.3-preview02 或更新版本
- MySQL 排它模式需要 8.0 以上版本
- TranLock 必須在事務中使用才會生效
- 查詢條件盡量用主鍵，減少鎖定範圍（避免表鎖）
- 多執行緒場景每個 Task 要 new 一個新的 db 實例，確保執行緒安全
- 悲觀鎖等待時間由資料庫的鎖等待逾時設定決定（SqlServer 預設無限等待，可設定 LOCK_TIMEOUT）
